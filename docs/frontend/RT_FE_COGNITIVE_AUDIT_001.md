## 1. 审计前提与方法说明

### 1.1 审计目标

回答核心问题：

> "一个**陌生但理性的人类**，在无背景说明的情况下打开 Run-Time 前端，
> 是否有任何可能误以为这是一个正在运行、正在工作的 AI 系统？"

### 1.2 审计方法

- **文档级审计**：仅审查 RT_FE_REQ_DRAFT.md 的描述性规范
- **认知视角**：从"陌生用户首次接触"的角度评估
- **风险维度**：五大错觉维度逐项检查
- **禁止事项**：不修改规范，不生成代码，不设计 UI

### 1.3 审计标准

**高风险（必须修正）**：
- 任何可能让用户认为"系统正在运行"的描述
- 任何类似 dashboard / monitor 的布局描述
- 任何暗示"实时数据"或"活动状态"的文案

**中风险（建议修正）**：
- 可能引起轻微误解的描述
- 不够明确的"只读"标识
- 可能被误读为"操作流程"的导航路径

**低风险（可接受）**：
- 需要上下文才能误解的描述
- 已有明确反误解护栏的描述

## 2. Run-Time 核心认知风险定义

### 2.1 禁止的认知错觉

以下错觉必须完全避免：

1. **执行错觉**：用户认为系统正在执行、运行、处理
2. **活动错觉**：用户认为有活动正在进行、数据在更新
3. **控制错觉**：用户认为这是一个控制台、可以操作
4. **监控错觉**：用户认为这是一个监控工具、显示实时状态
5. **智能错觉**：用户认为 AI 正在工作、系统在思考

### 2.2 允许的认知

以下认知是正确且期望的：

1. **阅读认知**：用户知道自己在阅读一个冻结的结构
2. **审计认知**：用户知道自己在检查一个不可变的快照
3. **参考认知**：用户知道这是一个用于引用的冻结记录
