## 0. 设计目标（不可更改）

本 Design-Time 前端的唯一目的，是帮助人类在【执行前】完成以下认知与判断：

- 构建一个 AI 虚拟公司的结构性声明
- 明确责任单元（Cell）及其边界
- 组合声明性拓扑（非执行流程）
- 选择或声明方法论视角
- 在人类确认后，将结构"冻结"为 Run-Time 可读但不可修改的状态

**本前端的本质**：
- 是结构设计工具
- 不是执行系统
- 不是控制台
- 不是自动化平台
- 不与 AI API 直接交互

**成功标准**：
人类在使用后应能清晰回答：
- "我设计的公司结构是什么？"
- "每个 Cell 的责任边界在哪里？"
- "我为什么要这样设计？"
- "我准备好冻结这个设计了吗？"

## 1. Design-Time 与 Run-Time 的严格区分

### 1.1 Design-Time 状态特征

**允许的操作**：
- 创建 Company 元信息
- 添加 / 编辑 / 删除 Cell
- 定义 Cell 责任描述与边界
- 附加 Role 约束到 Cell
- 构建声明性拓扑关系
- 选择或命名方法论视角
- 使用模板库
- 预览结构完整性
- 执行 Freeze 操作

**状态标识**：
- 界面顶部必须持续显示：【DESIGN-TIME】标识
- Company 显示状态：Draft
- 所有元素均可编辑
- 操作按钮使用编辑类词汇（编辑 / 删除 / 添加）

### 1.2 Run-Time 状态特征

**允许的操作**：
- 查看冻结后的结构
- 回溯设计决策
- 引用 Company 标识

**禁止的操作**：
- 任何编辑
- 任何执行
- 任何启动或停止

**状态标识**：
- 界面顶部显示：【RUN-TIME - READ ONLY】
- Company 显示状态：Frozen
- 所有编辑按钮消失或置灰
- 显示冻结时间戳与冻结人

### 1.3 Freeze：唯一的单向转换

**Freeze 的语义**：
- 这是一次人类承担责任的行为
- 表示"我确认这个设计可以作为执行依据"
- 是不可逆的单向操作

**Freeze 后的变化**：
- Draft 状态锁定为 Frozen
- 生成不可变的结构快照（包含完整 Cell / Role / Topology / Methodology）
- 提供只读的 Run-Time 引用 ID
- 原 Draft 不可再编辑

**如果需要修改**：
- 必须创建新的 Draft
- 可选择"基于 Frozen Company 创建新 Draft"
- 新旧版本通过 ID 关联，但完全独立

## 2. Design-Time 核心实体呈现规范

### 2.1 Company（设计中公司）

**实体定义**：
Company 是整个结构的语义锚点，用于回答"我在设计哪个虚拟公司"。

**必须呈现的字段**：
- Company 名称（可编辑文本输入框）
- Company 标识符（系统生成，只读，格式如 `COMP-{UUID}`）
- 人类可读描述（多行文本输入框）
- 当前状态（Draft / Frozen，只读标签）
- 创建时间（只读）
- 创建人（只读）

**呈现位置**：
- 界面顶部或左侧主导航区域
- 必须在所有界面中持续可见

**明确标识**：
- 必须显示 "DESIGN-TIME COMPANY" 文字标签
- Draft 状态使用中性色（非绿色，避免暗示"进行中"）

**禁止元素**：
- 启动 / 运行 / 执行按钮
- 进度指示器
- 健康度评分

---

### 2.2 Cell（责任单元）

**实体定义**：
Cell 是责任的最小声明单元。Cell 不是人，不是 AI，不是执行者，只是责任的结构性描述。

**必须呈现的字段**：

每个 Cell 必须包含：

1. **Cell 名称**（可编辑，简短标识）
2. **责任描述（What）**（可编辑，多行文本）
   - 说明：此字段回答"这个 Cell 负责什么"
3. **责任边界（What NOT）**（可编辑，多行文本）
   - 说明：此字段回答"这个 Cell 明确不负责什么"
4. **Cell 标识符**（系统生成，只读，格式如 `CELL-{UUID}`）
5. **附属 Role 约束列表**（见 2.3）

**呈现方式**：

**方式 A：列表视图**
- 以卡片或表格形式呈现所有 Cell
- 每个卡片显示：名称 / 责任摘要 / 边界摘要
- 点击进入详情编辑

**方式 B：网络图视图**
- 以节点形式呈现 Cell
- 节点内显示 Cell 名称
- 节点之间的连线表示声明性关系（见 2.4）
- 点击节点进入详情编辑

**操作允许**：
- 创建新 Cell（打开空白表单）
- 编辑现有 Cell（修改任何字段）
- 删除 Cell（需二次确认，并说明影响范围）
- 复制 Cell（创建副本）

**去拟人化要求**：
- 禁止使用头像图标
- 禁止使用"成员" "员工" "Agent"等词汇
- 禁止显示"执行状态"
- 界面文案使用"责任单元"而非"角色"或"成员"

**明确说明的事实**：
在 Cell 列表或详情页顶部，必须显示说明文字：
> "Cell 是责任声明，不是执行者。Cell 不会自动运行或分配给 AI。"

---

### 2.3 Role（约束）

**实体定义**：
Role 是 Cell 的附属约束描述，永远不能独立存在。Role 不是实体，不可被运行，不可被分配。

**必须呈现的字段**：
- Role 名称（可编辑）
- 约束类型（选择：允许 / 禁止 / 条件）
- 约束描述（可编辑，多行文本）

**呈现位置**：
- 必须在 Cell 详情页内呈现
- 以列表或折叠区域形式附属于 Cell
- 不可作为一级导航项

**操作允许**：
- 在 Cell 内添加 Role
- 编辑现有 Role
- 删除 Role

**禁止**：
- 单独创建 Role（必须先选择或创建 Cell）
- 将 Role 移动到其他 Cell（只能删除后重建）
- 执行 Role

**明确说明的事实**：
在 Role 区域顶部显示说明文字：
> "Role 是 Cell 的约束条件，不是独立实体。"

---

### 2.4 Workflow / Topology（声明性拓扑）

**实体定义**：
Topology 是 Cell 之间的声明性关系结构。它不是流程，不是步骤，不是执行路径。

**必须呈现的关系类型**：
1. **依赖关系**：Cell A 依赖 Cell B 的输出
2. **参考关系**：Cell A 参考 Cell B 的定义
3. **输入输出关系**：Cell A 的输出是 Cell B 的输入

**呈现方式**：

**方式 A：图形拓扑画布**
- 节点：Cell（显示名称）
- 边：关系类型（带文字标签）
- 布局：自动或手动调整
- 交互：
  - 拖拽 Cell 位置
  - 点击 Cell 进入详情
  - 点击边编辑关系类型
  - 右键或工具栏创建新关系

**方式 B：关系列表**
- 表格形式显示所有关系
- 列：源 Cell / 关系类型 / 目标 Cell / 描述

**操作允许**：
- 创建关系（选择源 Cell / 目标 Cell / 关系类型）
- 编辑关系（修改类型或描述）
- 删除关系

**严格禁止的元素**：
- 当前步骤标识
- 下一步高亮
- 执行路径动画
- 进度指示器
- 流程控制按钮（开始 / 暂停 / 停止）
- 任何颜色编码暗示"正在执行"

**明确说明的事实**：
在拓扑视图顶部显示说明文字：
> "这是声明性结构，不是执行流程。关系表示语义依赖，不表示时间顺序或执行路径。"

**去时态化要求**：
- 不使用"当前" "下一步" "正在"等词汇
- 不使用箭头暗示流向（使用无向线或双向箭头）
- 关系标签使用名词（依赖 / 参考）而非动词（执行 / 触发）

---

### 2.5 Methodology（方法论视角）

**实体定义**：
Methodology 是人类理解和阅读 Company 结构的认知视角。它不改变系统行为，不限制操作，不触发执行。

**必须呈现的字段**：
- 方法论名称（可编辑或选择）
- 方法论描述（可编辑，多行文本）
- 方法论状态（活跃 / 冻结 / 历史，只读）

**呈现位置**：
- 界面顶部或侧边栏
- 以下拉选择器或标签形式呈现
- 切换后不刷新页面，仅改变视角

**切换效果**：
当切换方法论时，允许以下变化：
- Cell 的分组方式改变（如按阶段分组 vs 按职能分组）
- 拓扑的布局方式改变（如层级结构 vs 网状结构）
- 标注或高亮某些关系（但不隐藏其他关系）

**不允许的变化**：
- 不改变 Cell 的数据内容
- 不创建或删除 Cell
- 不改变关系的实际定义
- 不触发任何后端行为

**操作允许**：
- 选择预定义方法论（如 DevOps / Agile / Functional）
- 创建自定义方法论
- 组合多个方法论

**禁止元素**：
- 方法论推荐或评分
- 冲突警告
- 自动切换
- 方法论驱动的操作限制

**明确说明的事实**：
在方法论选择器旁显示说明文字：
> "方法论是理解视角，不改变系统行为。"

---

### 2.6 Template Library（模板库）

**实体定义**：
Template 是预定义的 Company 结构模板，用于快速启动设计。

**必须呈现的内容**：
- 模板名称
- 模板描述
- 包含的 Cell 数量
- 包含的关系数量
- 预设的方法论

**操作允许**：
- 浏览模板列表
- 预览模板结构（只读）
- 基于模板创建新 Draft（复制模板内容到新 Company）

**禁止操作**：
- 修改系统预定义模板
- 自动应用模板
- 模板推荐或匹配

**明确说明的事实**：
在模板库顶部显示说明文字：
> "模板是起点，不是约束。选择模板后，你可以自由修改所有内容。"

## 3. Design-Time 信息层级与导航

### 3.1 信息层级定义

```
L1: Company Context（公司整体）
    ├─ Company 元信息
    ├─ 当前状态标识
    └─ Freeze 操作入口

L2: Structure Layer（结构层）
    ├─ Cell 列表视图
    ├─ Cell 详情编辑
    └─ Role 附属信息

L3: Topology Layer（拓扑层）
    ├─ 声明性拓扑画布
    └─ 关系列表

L4: Methodology View（方法论视角）
    ├─ 方法论选择器
    └─ 视角说明

L5: Template Library（模板库）
    ├─ 模板浏览
    └─ 模板预览

L6: Freeze Confirmation（冻结确认）
    ├─ 结构完整性摘要
    └─ 冻结二次确认
```

### 3.2 导航结构

**主导航（始终可见）**：
1. Company Overview（L1）
2. Cell Management（L2）
3. Topology Canvas（L3）
4. Template Library（L5）

**全局控制（右上角或底部）**：
1. Methodology Selector（L4，下拉选择器）
2. Freeze Design（L6，主要操作按钮）

**二级导航（上下文进入）**：
- 从 Cell 列表点击进入 Cell 详情
- 从 Cell 详情查看附属 Role
- 从 Topology 点击节点进入 Cell 详情

### 3.3 每层允许 / 禁止的操作

| 层级 | 允许的操作 | 禁止的操作 |
|------|-----------|-----------|
| L1 | 编辑 Company 元信息 | 启动 / 运行 Company |
| L2 | 创建 / 编辑 / 删除 Cell | 执行 Cell / 分配给 AI |
| L3 | 创建 / 编辑 / 删除关系 | 执行流程 / 当前步骤标识 |
| L4 | 切换方法论视角 | 方法论驱动操作限制 |
| L5 | 浏览和使用模板 | 修改系统模板 |
| L6 | 确认 Freeze | 编辑已 Freeze 的内容 |

## 4. 人类的主要设计路径（认知流程）

### PATH A：从零创建公司结构

**步骤 1：创建 Company**
- 人类理解：我要给这个虚拟公司命名并描述其目的
- 界面呈现：空白 Company 表单
- 人类操作：填写名称和描述
- 系统反馈：保存成功，显示 Draft 标识

**步骤 2：定义第一个 Cell**
- 人类理解：我要定义第一个责任单元
- 界面呈现：Cell 创建表单
- 人类操作：填写责任描述和边界
- 系统反馈：Cell 出现在列表中

**步骤 3：添加更多 Cell**
- 人类理解：我要构建完整的责任网络
- 界面呈现：重复步骤 2
- 人类操作：逐个添加 Cell
- 系统反馈：Cell 列表增长

**步骤 4：定义 Cell 之间的关系**
- 人类理解：我要说明 Cell 之间的依赖和参考关系
- 界面呈现：拓扑画布或关系创建表单
- 人类操作：选择源 Cell / 目标 Cell / 关系类型
- 系统反馈：关系线出现在拓扑图中

**步骤 5：选择方法论视角**
- 人类理解：我要用某种视角组织和理解这个结构
- 界面呈现：方法论选择器
- 人类操作：选择或创建方法论
- 系统反馈：界面布局或分组方式调整

**步骤 6：审查完整性**
- 人类理解：我要检查是否有遗漏或错误
- 界面呈现：完整性摘要（Cell 数量 / 关系数量 / 孤立 Cell 警告）
- 人类判断：是否准备好冻结
- 系统反馈：仅展示摘要，不做评价

**步骤 7：Freeze**
- 人类理解：我确认这个设计可以作为执行依据
- 界面呈现：Freeze 确认对话框（见 5.2）
- 人类操作：二次确认
- 系统反馈：状态变为 Frozen，所有编辑能力消失

---

### PATH B：基于模板修改并定制

**步骤 1：浏览模板库**
- 人类理解：我想找一个接近我需求的起点
- 界面呈现：模板列表，带名称和描述
- 人类操作：浏览并预览模板
- 系统反馈：展示模板的结构预览

**步骤 2：选择模板并创建 Draft**
- 人类理解：我要基于这个模板开始
- 界面呈现："使用此模板"按钮
- 人类操作：点击按钮
- 系统反馈：创建新 Draft，内容复制自模板

**步骤 3：修改 Cell**
- 人类理解：我要调整某些责任单元以适应我的需求
- 界面呈现：可编辑的 Cell 列表
- 人类操作：编辑或删除 Cell
- 系统反馈：内容更新

**步骤 4：调整拓扑**
- 人类理解：我要改变某些关系
- 界面呈现：可编辑的拓扑画布
- 人类操作：添加 / 删除 / 修改关系
- 系统反馈：拓扑更新

**步骤 5-7：同 PATH A**

---

### PATH C：审查现有 Draft 并准备 Freeze

**步骤 1：打开现有 Draft**
- 人类理解：我要继续之前未完成的设计
- 界面呈现：Draft 列表，显示创建时间和修改时间
- 人类操作：选择并打开 Draft
- 系统反馈：进入该 Draft 的编辑状态

**步骤 2：系统化审查 Cell**
- 人类理解：我要逐一检查每个 Cell 的责任和边界是否清晰
- 界面呈现：Cell 列表，支持筛选和排序
- 人类操作：逐个点击查看详情
- 系统反馈：展示完整 Cell 信息

**步骤 3：检查拓扑完整性**
- 人类理解：我要确认关系是否合理
- 界面呈现：拓扑画布，可能显示孤立节点警告
- 人类操作：查看并调整
- 系统反馈：仅提示事实（如"3 个 Cell 无任何关系"），不做评价

**步骤 4：确认方法论**
- 人类理解：我要确认当前的方法论视角是否符合预期
- 界面呈现：当前方法论标识
- 人类操作：切换或保持
- 系统反馈：视角调整

**步骤 5：Freeze**
- 同 PATH A 步骤 7

## 5. Freeze 机制（关键）

### 5.1 Freeze 的设计语义

Freeze 是本系统中最重要的人类判断点。它的语义是：

> "我作为设计者，确认这个结构性声明已经足够清晰和完整，可以作为后续执行的依据。我承担这个设计的责任。"

**Freeze 不是**：
- 自动校验通过
- AI 推荐的结果
- 成熟度达标

**Freeze 是**：
- 人类的明确承诺
- 不可逆的决策
- 责任的锚定点

### 5.2 Freeze 前的展示内容

当人类点击"Freeze Design"按钮后，必须显示以下内容：

**对话框标题**：
"确认冻结设计"

**摘要信息（只读）**：
1. Company 名称和描述
2. Cell 总数
3. 关系总数
4. 当前采用的方法论
5. 孤立 Cell 数量（如有）
6. 无边界声明的 Cell 数量（如有）

**警告信息（如适用）**：
- "有 3 个 Cell 未定义责任边界（What NOT）"
- "有 2 个 Cell 未与其他 Cell 建立任何关系"

注意：这些是事实陈述，不是错误提示。不阻止 Freeze。

**确认文字（必须显示）**：
> "冻结后，此设计将变为只读状态，不可再编辑。如需修改，只能创建新的 Draft。"

**二次确认机制**：
- 显示文本输入框，要求人类输入 Company 名称以确认
- 或：显示复选框"我确认已审查所有内容并承担设计责任"

**操作按钮**：
- "取消"（次要按钮）
- "确认冻结"（主要按钮，危险色或强调色）

### 5.3 Freeze 后的状态变化

**立即变化**：
1. Company 状态从 Draft 变为 Frozen
2. 所有编辑按钮消失或置灰
3. 显示冻结时间戳
4. 显示冻结人标识
5. 生成只读引用 ID（如 `FROZEN-COMP-{UUID}`）

**界面变化**：
1. 顶部标识从"DESIGN-TIME"变为"RUN-TIME - READ ONLY"
2. 主导航移除"编辑"相关项
3. 所有表单字段变为只读文本
4. Freeze 按钮消失

**提供的新能力**：
1. 导出结构快照（JSON 或 YAML）
2. 复制为新 Draft（创建可编辑副本）
3. 查看冻结历史记录

### 5.4 Freeze 的撤销与版本管理

**明确规则**：
- Freeze 不可撤销
- Frozen Company 不可回退到 Draft
- 如需修改，必须创建新 Draft

**版本关联**：
- 新 Draft 可标注"基于 FROZEN-COMP-{UUID} 创建"
- 系统保留版本链，但不强制版本控制
- 人类可查看历史版本，但不能合并或回滚

## 6. 明确禁止的 Design-Time 语义

以下元素或语义在 Design-Time 前端中必须完全禁止：

### 6.1 执行类语义

禁止的操作：
- 启动 / 运行 / 执行 Company
- 启动 / 分配 / 触发 Cell
- 模拟 / 测试执行
- 开始 / 暂停 / 停止流程

禁止的界面元素：
- "Start" / "Run" / "Execute" 按钮
- 播放 / 暂停 / 停止图标
- 进度条
- 当前步骤指示器
- 执行日志或输出窗口

### 6.2 AI 辅助类语义

禁止的操作：
- AI 自动生成 Cell
- AI 推荐拓扑结构
- AI 评分或优化建议
- 智能补全或自动填充（基于 AI）

禁止的界面元素：
- "AI 助手"按钮
- "智能推荐" / "优化建议"
- "自动生成"功能
- AI 对话框或聊天窗口

### 6.3 评价类语义

禁止的操作：
- 结构评分（如"成熟度 85%"）
- 风险评估
- 最佳实践匹配度
- 健康度检查

禁止的界面元素：
- 评分表盘
- 健康度指示器（绿 / 黄 / 红）
- KPI 仪表盘
- "优秀" / "良好" / "需改进"标签

### 6.4 混合状态语义

禁止的行为：
- 在 Design-Time 界面显示 Run-Time 数据
- 在 Run-Time 界面允许编辑
- 同时展示 Draft 和 Frozen 状态的 Company
- 从 Run-Time 直接编辑跳转到 Design-Time

明确要求：
- Design-Time 和 Run-Time 必须是完全独立的界面模式
- 切换必须显式且不可逆（通过 Freeze）

### 6.5 隐式代理类语义

禁止的行为：
- 系统代替人类做任何决策
- 默认选项或推荐项
- 自动保存后提示"已为你优化"
- 任何暗示"系统在帮你工作"的文案

明确要求：
- 所有操作必须由人类显式触发
- 所有保存必须明确反馈"已保存你的输入"
- 不使用"为你" / "帮你" / "自动"等词汇

## 7. 界面视觉与交互原则

### 7.1 视觉定位

**设计参考**：
- CAD 软件（结构性 / 精确性）
- 建筑设计工具（蓝图感 / 规划感）
- 思维导图工具（关系性 / 清晰性）

**避免的参考**：
- 现代 SaaS 仪表盘（避免暗示自动化）
- 项目管理工具（避免暗示执行）
- AI 产品界面（避免暗示智能）

### 7.2 色彩原则

**主色调**：
- 中性色为主（灰 / 深蓝 / 浅灰）
- 避免绿色（暗示成功或运行中）
- 避免红色（暗示错误或警告）

**强调色**：
- Freeze 按钮：使用强调色（如深橙或深蓝）
- 编辑状态：使用中性高亮（如浅蓝边框）

**状态色**：
- Draft：中性蓝
- Frozen：深灰或锁定图标

### 7.3 文案原则

**动词选择**：
- 使用：定义 / 描述 / 声明 / 添加 / 编辑 / 删除
- 避免：执行 / 运行 / 触发 / 启动 / 分配 / 完成

**名词选择**：
- 使用：责任单元 / 声明 / 结构 / 关系
- 避免：角色 / 成员 / 任务 / 流程 / 步骤

**说明文字特征**：
- 客观陈述事实
- 不使用第二人称"你应该"
- 不使用评价性词汇"优秀" / "差"

### 7.4 交互速度

**响应延迟**：
- 编辑操作：即时反馈
- 保存操作：0.5-1 秒延迟（人为可感知的停顿）
- Freeze 操作：2 秒延迟（强调严肃性）

**目的**：
- 避免暗示系统在"快速自动化"
- 给予人类思考时间
- 强化"设计是严肃行为"的感知

### 7.5 确认机制

**需要二次确认的操作**：
- 删除 Cell
- 删除关系
- Freeze Company
- 删除整个 Draft

**确认方式**：
- 对话框 + 文字说明
- 输入确认文本（如 Cell 名称）
- 或：复选框 + 明确确认按钮

### 8.2 假设的后端 API（不由前端实现）

前端需要调用的 API 端点（假设存在）：

**Company**：
- `POST /api/design-time/companies` - 创建 Draft
- `GET /api/design-time/companies/:id` - 获取 Draft
- `PUT /api/design-time/companies/:id` - 更新 Draft
- `POST /api/design-time/companies/:id/freeze` - 冻结

**Cell**：
- `POST /api/design-time/companies/:companyId/cells` - 创建 Cell
- `PUT /api/design-time/cells/:id` - 更新 Cell
- `DELETE /api/design-time/cells/:id` - 删除 Cell

**Topology**：
- `POST /api/design-time/companies/:companyId/relations` - 创建关系
- `DELETE /api/design-time/relations/:id` - 删除关系

**Template**：
- `GET /api/design-time/templates` - 获取模板列表
- `POST /api/design-time/templates/:id/use` - 基于模板创建 Company
